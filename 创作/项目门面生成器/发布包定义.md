# 发布包定义

这份文档不负责生成 README、帖子或视觉素材。

它的作用是把这些分散产物收成一个统一的交付对象：

**发布包**

一句话理解：

`不是交一个文件，而是交一包能发出去的材料。`

---

## 发布包是什么

发布包是某个项目、skill、工具或作品在“准备对外发布”时的最小交付集合。

它不等于：

- 只有一份 README
- 只有一条帖子
- 只有一张图

它更像一套对外材料包，至少要回答：

- 这是什么
- 对谁有用
- 怎么进入
- 用什么入口发出去
- 需要哪些视觉门面
- 是否已经具备正式发布条件

---

## 发布包的作用

定义发布包，主要是为了解决 4 个问题：

1. 让“整理门面”和“发出去”不再是两件脱节的事
2. 让上游 skill 有明确交付对象，而不是只交一个 README
3. 让下游 skill 有统一输入，而不是每次临时拼
4. 让发布是否完成，有一个可以检查的边界

---

## 最小发布包

最小发布包适合：

- 先发一个项目
- 先给别人看
- 先形成最基本的对外入口

最小发布包包括：

| 项目 | 说明 | 必填 |
| --- | --- | --- |
| `README` | 基础项目门面 | 是 |
| `一句话定位` | 快速说明这是什么 | 是 |
| `主链接` | 仓库或文档主入口 | 是 |
| `最小发帖文案` | 主帖或一句对外介绍 | 是 |
| `推荐截图` | 至少一张能展示的图 | 是 |

一句话理解：

**最小发布包 = 能发出去，但不一定完整。**

---

## 标准发布包

标准发布包适合：

- 已经要正式发给别人看
- 不只是“先发一下”
- 希望别人能真正看懂、点进来、继续用

标准发布包包括：

| 项目 | 说明 | 必填 |
| --- | --- | --- |
| `README` | 完整项目门面 | 是 |
| `安装 / 使用方式` | 让别人能进入 | 是 |
| `主链接` | 仓库、主页或文档入口 | 是 |
| `推荐截图` | 展示内容的主图 | 是 |
| `主帖` | 对外主发布文案 | 是 |
| `第一条带链接跟帖` | 分发时的直接入口 | 是 |
| `一句话定位` | 全链路统一锚点 | 是 |
| `核心亮点` | 最多 3 个核心卖点 | 是 |

一句话理解：

**标准发布包 = 能看、能点、能理解、能转发。**

---

## 正式发布包

正式发布包适合：

- 准备认真对外发布
- 希望门面、发帖和视觉是一致的
- 希望后续还能继续扩平台

正式发布包包括：

| 项目 | 说明 | 必填 |
| --- | --- | --- |
| `README` | 已通过 README 评测 | 是 |
| `主链接` | 主入口明确 | 是 |
| `推荐截图` | 有可直接对外展示的图 | 是 |
| `主帖 + 跟帖` | 至少一组完整发帖文案 | 是 |
| `交接底稿` | 供下游 skill 继续消费 | 是 |
| `LICENSE` | 若当前项目需要 | 按需 |
| `视觉门面` | SVG 标题 / 横幅 / 信息卡等 | 按需 |
| `Git 收尾状态` | 是否已提交 / 已推送 | 按需 |

一句话理解：

**正式发布包 = 不只是能发，而是发出去时材料一致、边界清楚。**

---

## 谁负责产出什么

### 项目门面生成器

优先负责：

- `README`
- `一句话定位`
- `核心亮点`
- `安装 / 使用方式`
- `主链接`
- `推荐截图`
- `LICENSE`（按策略）
- `视觉门面`（仅在明确视觉意图或已有素材时）
- `Git 收尾状态`

### 发帖类 skill

优先负责：

- `主帖`
- `第一条带链接跟帖`
- `补充跟帖`

### 视觉类 skill

优先负责：

- `信息卡`
- `SVG 标题`
- `SVG 横幅`
- 动画或其他视觉门面素材

---

## 发布包完成条件

### 最小发布包完成

以下全部满足时，可视为完成：

- README 可读
- 有一句话定位
- 有主链接
- 有一条能发出去的话
- 有至少一张推荐截图

### 标准发布包完成

以下全部满足时，可视为完成：

- README 评测为绿色
- 安装 / 使用方式明确
- 主帖和第一条跟帖已完成
- 推荐截图明确
- 核心亮点不超过 3 个且可复述

### 正式发布包完成

以下全部满足时，可视为完成：

- README 评测为绿色
- 发帖产物已通过对应评测
- 交接字段协议已补齐最低字段
- 若需要视觉门面，则视觉产物已通过对应评测
- 若需要 LICENSE，则 LICENSE 已明确
- 若需要 Git 收尾，则提交边界明确

---

## 最低放行问题

任何发布包，在放行前都至少要过这 5 个问题：

1. 我能不能一句话说清这是什么？
2. 别人看完第一屏，会不会知道这是不是给他用的？
3. 我有没有明确入口让别人点进去？
4. 我有没有至少一张图或一个视觉锚点？
5. 这次发布到底停在“先发一下”，还是“正式发出”？

---

## 和交接字段协议的关系

- `交接字段协议` 定义“信息怎么交接”
- `各产物评测标准` 定义“产物怎么过审”
- `发布包定义` 定义“最后到底交什么”

这三者是同一套东西的三层：

- 字段
- 评测
- 交付

---

## 一句话结论

发布包的意义，不是让你多做几份材料，而是：

**让每次发布都有一个明确的交付边界。**
