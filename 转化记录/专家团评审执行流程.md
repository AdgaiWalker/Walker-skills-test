# 通用工作流模板

> 基于 iFlow CLI + OpenClaw 的任务执行框架

---

## 核心提示词

```
【任务类型】对 {目标} 进行 {动作}

【专家团】{维度1}、{维度2}、{维度3}

【输出】
- 汇总问题，按严重度分级
- 生成方案：{目标}/方案.md

【执行】交给OpenClaw执行，验证修复，只给结果
```

---任务传递的话，就是说我给你说一下我们的工作流。就是说一共有 3 个维度。

如果就是说，我自己去当老大，然后 I Follow 是老二，Open Cloud 是老三。老大也就是我，我来制定目标，做什么和不做什么，也是有标准的。

然后 I Follow 就是说根据老大的目标来进行一个规划。比如说调查一些资料，然后做一个专家团，让这些专家去干什么什么的，也就是我们上面那样，然后做出一个安全的、优质的这么一份规划吧。

哦，还忘了就是说不仅有专家还有各种就是说各界的用户，各界的专家，各界的用户，然后就是轮流轮流去验证。比如说提，然后就是 Open Cloud 的话，Open Cloud 就是说根据比如说老大的旨意，然后老二的规划，然后去进行执行。然后执行过程中可能会出错，出完错之后咱们也不用担心，假如说错了，然后老二还有一个职位，就是说老二要去监督这个老三。老三做错的事老二知道了，老二知道就是说去分析哪里出错了，看看他哪里出错了。然后老三每做一个就要去验证，看有没有出错。然后再去进行下一个任务。然后老二的话就负责监督，比如说在 Open Cloud，老三在做完所有的任务之后，然后再去核验一遍到底有没有错。就说哪里有没有做错，哪里有遗漏，哪里没有做好。然后老三负责监督 Open Cloud。老二监督老三干活。然后最后结果出来，老大也就是我，我自己比如说去验收这个结果。如果满意的话那就行，而如果不满意，比如说有问题，我让他再进行一次轮回。这是我的想法。第四个权限处理涉及到文件信息的安全，以及防止本地文件被删除。

第二个是防止泄露。那么如何防止删除呢？我就借用 GitHub 和 Git。本地仓库用 Git，远程仓库用 GitHub 创建远程仓库。

然后通过一些命令，比如让 Open Cloud 做一个文件改动。比如以文件为真，为第一要性。一旦改动，就自动进行本地仓库与远程仓库的一个同步。

然后 Open Cloud 执行的话，我说过了老大、老二、老三。老大可以管老二、老三；老二也可以管老二、老三。所以说这个权限的分发以老大为大。

但是呢，也就是说老二、老三在执行的时候，就让老二能管老三，直接让老二这么批准就行了



## 任务类型速查

| 类型        | 目标   | 动作   | 专家维度            |
| --------- | ---- | ---- | --------------- |
| **代码评审**  | 项目路径 | 全面评审 | UI/UX、前端、后端、SEO |
| **安全审计**  | 项目路径 | 安全检查 | 后端安全、认证、数据保护    |
| **性能优化**  | 项目路径 | 性能分析 | 前端性能、后端性能、数据库   |
| **重构**    | 模块路径 | 代码重构 | 架构、设计模式、可维护性    |
| **新功能**   | 需求描述 | 设计实现 | 架构、UI/UX、技术选型   |
| **Bug修复** | 问题描述 | 根因分析 | 相关技术专家          |

## 环境配置

### OpenClaw 启动

```powershell

schtasks /Change /TN "OpenClaw Gateway" /ENABLE
schtasks /Run /TN "OpenClaw Gateway"

# 访问
start http://127.0.0.1:18789/
```

## 核心原则

1. **问题分级** - 严重 > 高 > 中 > 低
2. **先分析后执行** - 生成方案再交给OpenClaw
3. **验证闭环** - 每个任务完成都要验证
4. **只给结果** - 不中间汇报，减少噪音
