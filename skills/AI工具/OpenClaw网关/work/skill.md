# OpenClaw Commander - 三层代理架构

## 核心原则

**目标从上往下流，结果从下往上回，状态写在中间文件里。**

---

## 一、架构概览

```
老大（用户）：定目标 + 定标准
老二（iFlow）：规划 + 分派 + 汇总
老三（OpenClaw）：执行 + 写状态
```

## 三、层级职责

| 层级 | 角色 | 职责 | 输入 | 输出 |
|------|------|------|------|------|
| 第一层 | 老大 | 定目标、定标准、验收 | 需求 | 目标文档、验收标准 |
| 第二层 | 老二 | 规划、分派、汇总 | 目标文档 | 任务分配、汇总报告 |
| 第三层 | 老三 | 执行、写状态 | 任务分配 | 结果文件 |

---

## 四、激活时机

当用户请求以下操作时，自动激活此技能：
- "把这个任务交给 OpenClaw 执行"
- "让 OpenClaw 去做..."
- "委派给 OpenClaw"
- 用户提供了明确的执行方案，需要交给 OpenClaw 落地

---

## 五、CLI 通信协议

### 5.1 派发任务（老二 → 老三）

```bash
# 派发任务，指定结果文件路径
openclaw agent --agent main --message "
## 任务：xxx

完成后将结果写入 status/a.json

格式：
{
  \"task\": \"任务名称\",
  \"status\": \"done\",
  \"output\": \"执行结果\"
}
" --json
```

### 5.2 查看会话状态

```bash
# 列出所有会话
openclaw sessions --json

# 查看活跃会话（最近2小时）
openclaw sessions --active 120 --json
```

### 5.3 检查 Gateway 状态

```bash
openclaw status
```

## 六、并行执行

### 6.1 同时派发多个任务

```bash
# 任务A
openclaw agent --agent main --message "任务A：优化前端，结果写入 status/a.json" --json

# 任务B
openclaw agent --agent main --message "任务B：优化后端，结果写入 status/b.json" --json

# 任务C
openclaw agent --agent main --message "任务C：优化数据库，结果写入 status/c.json" --json
```

### 6.2 结果文件目录结构

```
status/
├── a.json  ← 老三-A 的结果
├── b.json  ← 老三-B 的结果
└── c.json  ← 老三-C 的结果
```

---

## 七、结果文件格式

```json
{
  "task": "任务名称",
  "status": "done | failed | running",
  "startTime": "2026-03-01T10:00:00",
  "endTime": "2026-03-01T10:30:00",
  "output": "执行结果摘要",
  "progress": "100%",
  "error": null,
  "dependsOn": null
}
```

## 八、状态流转

```
┌─────────┐     执行      ┌─────────┐     完成      ┌─────────┐
│ pending │ ───────────→ │ running │ ───────────→ │  done   │
└─────────┘              └────┬────┘              └─────────┘
                              │
                              │ 失败
                              ▼
                         ┌─────────┐
                         │ failed  │
                         └─────────┘
```

---

## 九、异常处理

| 异常类型 | 检测方式 | 处理方式 |
|----------|----------|----------|
| 失败 | status: failed | 老二报告老大，决定重试或跳过 |
| 卡住 | startTime 超时未完成 | 老二报告老大 |
| 冲突 | 多个老三写同一文件 | 老二统一合并，各老三写各自文件 |
| 依赖 | dependsOn 文件未 done | 老二调度顺序 |
| 进度 | progress 字段 | 老二随时读取 |

## 十、执行流程

### Step 1: 接收目标（来自老大）
1. 明确任务目标
2. 确认验收标准

### Step 2: 规划任务
1. 拆分为多个子任务
2. 确定依赖关系
3. 分配结果文件路径

### Step 3: 检查环境
```bash
openclaw status
```

### Step 4: 并行派发
```bash
# 每个任务指定独立的结果文件
openclaw agent --agent main --message "任务A，结果写入 status/a.json" --json
openclaw agent --agent main --message "任务B，结果写入 status/b.json" --json
```

### Step 5: 汇总结果
1. 读取所有结果文件
2. 检查状态（done/failed）
3. 生成汇总报告

### Step 6: 报告老大
- 展示汇总结果
- 标注失败任务
- 提供下一步建议

## 十一、Token 优化策略

| 方式 | Token 消耗 | 推荐度 |
|------|-----------|--------|
| 老二读会话记录 | 高（全部历史） | 不推荐 |
| 老三写结果文件 | 低（仅摘要） | 推荐 |

**原则：老三写摘要，老二读摘要。不传对话历史，只传结构化结果。**

---

## 十二、汇报格式

```
📋 任务汇总报告

┌─────────────┬────────────┬──────────┐
│ 任务        │ 状态       │ 结果     │
├─────────────┼────────────┼──────────┤
│ 任务A       │ ✅ done    │ xxx      │
│ 任务B       │ ❌ failed  │ 权限不足 │
│ 任务C       │ ⏳ running │ 50%      │
└─────────────┴────────────┴──────────┘

建议：
- 任务B 需要提升权限后重试
- 任务C 进行中，等待完成
```

---

## 十三、注意事项

1. **各写各的文件**：多个老三不能写同一个结果文件
2. **状态要写全**：包含 status、startTime、output 等字段
3. **异常要上报**：失败/卡住要报告老大，不要隐瞒
4. **依赖要检查**：有依赖关系的任务，老二负责调度顺序

---

## 一句话总结

> **老三分头写状态，老二读状态汇总，出事报老大。**
