# 三层代理架构设计报告

## 一、架构概览

```
老大（用户）：定目标 + 定标准
老二（iFlow）：规划 + 分派 + 汇总
老三（OpenClaw）：执行 + 写状态
```

**核心原则：目标从上往下流，结果从下往上回，状态写在中间文件里。**

## 三、层级职责

| 层级 | 角色 | 职责 | 输入 | 输出 |
|------|------|------|------|------|
| 第一层 | 老大（用户） | 定目标、定标准、验收 | 需求 | 目标文档、验收标准 |
| 第二层 | 老二（iFlow） | 规划、分派、汇总 | 目标文档 | 任务分配、汇总报告 |
| 第三层 | 老三（OpenClaw） | 执行、写状态 | 任务分配 | 结果文件 |

---

## 四、通信机制

### 4.1 老二 → 老三（CLI 派发）

```bash
# 派发任务
openclaw agent --agent main --message "任务内容，结果写入 status/a.json" --json
```

### 4.2 老三 → 老二（结果文件）

```
status/
├── a.json  ← 老三-A 的结果
├── b.json  ← 老三-B 的结果
└── c.json  ← 老三-C 的结果
```

### 4.3 结果文件格式

```json
{
  "task": "任务名称",
  "status": "done | failed | running",
  "startTime": "2026-03-01T10:00:00",
  "endTime": "2026-03-01T10:30:00",
  "output": "执行结果摘要",
  "progress": "100%",
  "error": null,
  "dependsOn": null
}
```

```
                              │
                              │ 失败
                              ▼
                         ┌─────────┐
                         │ failed  │
                         └─────────┘
```

## 六、异常处理

| 异常类型 | 检测方式 | 处理方式 |
|----------|----------|----------|
| 失败 | status: failed | 老二报告老大，决定重试或跳过 |
| 卡住 | startTime 超时未完成 | 老二报告老大 |
| 冲突 | 多个老三写同一文件 | 老二统一合并，各老三写各自文件 |
| 依赖 | dependsOn 文件未 done | 老二调度顺序 |
| 进度 | progress 字段 | 老二随时读取 |

---

## 七、完整流程

```
1. 老大定目标、定标准
       ↓
2. 老二规划任务，拆分为多个子任务
       ↓
3. 老二通过 CLI 并行派发任务给多个老三
       ↓
4. 老三各自执行，写入结果文件（含状态）
       ↓
5. 老二读取所有结果文件，汇总报告
       ↓
6. 老大验收报告
       ↓
   通过 → 结束
   返工 → 回到步骤 2
```

---

## 八、Token 优化策略

| 方式 | Token 消耗 | 推荐度 |
|------|-----------|--------|
| 老二读会话记录 | 高（全部历史） | 不推荐 |
| 老三写结果文件 | 低（仅摘要） | 推荐 |

**原则：老三写摘要，老二读摘要。不传对话历史，只传结构化结果。**

---

## 九、一句话总结

> **老三分头写状态，老二读状态汇总，出事报老大。**
