# 案例库

这份文档不是再讲方法论，而是拿真实项目验证这套方法怎么落地。

它的作用是：

- 把抽象规则压回真实项目
- 看这套工作流到底产出了什么
- 记录哪些地方已经跑通，哪些地方还没跑通

一句话理解：

`案例库负责证明这套方法不是空话。`

---

## 使用方式

看案例时，优先看这 6 个问题：

1. 原始状态是什么
2. 这次目标是什么
3. 交接字段有没有补齐
4. 最后产出了什么
5. 评测结果是什么
6. 下一步还差什么

---

## 案例 1：项目门面生成器

对象：

- [项目门面生成器 README](</E:/桌面/skills测试台/Skills-Walker/创作/项目门面生成器/README.md>)
- [项目门面生成器 skill](</E:/桌面/skills测试台/Skills-Walker/创作/项目门面生成器/skill.md>)

### 原始状态

- 一开始更像 README 生成器
- 后来逐步扩到了 LICENSE、Git 收尾、正式发布、视觉门面
- 但在中间阶段，方法论增长速度一度快过实际落地速度

### 这次目标

- 把它从“会讲方法”的 skill 收成“能交付发布包”的 skill
- 明确它不是只写 README，而是整理项目门面
- 明确正式发布时视觉素材如何进入链路

### 关键交接字段

- 项目名：项目门面生成器
- 一句话定位：把项目整理到能见人、能介绍、能移交的状态
- 解决什么问题：项目做出来了，但 README、安装、发布门面不完整
- 对谁有用：独立开发者、AI 编程用户、开源作者、skill/工具开发者
- 当前发布目标：让别人快速理解并直接调用这个 skill

### 产出结果

- 人看的 README
- 给 AI 的 skill 规则
- 安装方式
- README 方法论总纲
- 交接字段协议
- 各产物评测标准
- 发布包定义
- 平台扩展框架
- 风格卡

### 当前评测

- README：绿色
- 方法文档：绿色
- 执行能力：黄色

黄色的原因：

- 方法底盘已经很完整
- 但还需要继续把这些文档压回 skill 的执行合同和真实交付结果里

### 下一步优化

- 继续把方法文档压回执行规则
- 明确“完成定义”和“输出合同”
- 继续用真实项目而不是新文档验证

## 案例 2：creator-publish-workflow

对象：

- [Walker-skills-test 仓库](https://github.com/AdgaiWalker/Walker-skills-test)

### 原始状态

- 最初只是从 `Skills-Walker` 中抽出两个 skill
- 一开始更像“创作者发布工作流”
- 后来发现如果放在 GitHub 语境里，这种说法会显得偏内容侧

### 这次目标

- 做成一个主题仓库，而不是技能大合集
- 把“做出来 -> 整理出来 -> 发出去”收成一条清楚的链
- 补上视觉支持工具

### 关键交接字段

- 项目名：creator-publish-workflow
- 一句话定位：把作品从做出来，推进到整理好、发出去
- 核心亮点：
  - 项目门面生成器
  - 秋知 X 推文发帖生成器
  - 信息卡截图生成器
  - SVG组装动画器
- 当前发布目标：作为主题发布仓库，对外展示这一套工作流

### 产出结果

- 主题仓库首页
- 两个核心 skill
- 两个视觉支持工具
- 安装入口
- 方法底稿文档

### 当前评测

- 仓库首页：绿色
- 主链清晰度：绿色
- 仓库命名和定位：黄色

黄色的原因：

- “creator” 这个词在 GitHub 上仍偏内容创作者语境
- 长期看更适合往“project publish / distribution”语义收

### 下一步优化

- 继续收命名和第一屏定位
- 让主题仓库和实验区之间的同步规则更机械
- 逐步把平台扩展框架映射到主题仓库

## 案例 3：Ferry

对象：

- [Ferry 仓库](https://github.com/AdgaiWalker/Ferry)

### 原始状态

- README 原始内容偏宣言和理论表达
- 对理论 / 方法论读者有吸引力
- 但对第一次进来的人，入口层不够快

### 这次目标

- 测试门面 skill 能不能处理理论 / 方法论项目
- 判断“理论感”和“入口感”之间怎么平衡
- 补一轮正式发布排版和视觉实验

### 关键交接字段

- 项目名：Ferry
- 一句话定位：面向人机协作的世界协议
- 为什么做：连接意图、执行、记忆与进化
- 解决什么问题：对抗断裂、熵增和遗忘
- 对谁有用：方法论爱好者、人机协作探索者、复杂系统工作者
- 推荐截图：README 首页 / Ferry 信息卡 / SVG 标题视觉

### 产出结果

- 多轮 README 调整和回退
- 一版正式发布排版
- 一张小红书信息卡
- 一版可组装的 SVG 标题素材

### 当前评测

- README：黄色
- 信息卡：绿色
- SVG 门面：黄色

黄色的原因：

- README 的“理论气质”和“入口速度”还在拉扯
- SVG 已经有静态门面价值，但动画页还没完全收稳

### 下一步优化

- 不再反复大改 README 立场，只精调第一屏
- 让信息卡和 README 的语气继续统一
- 继续把 SVG 从“能看”推进到“可稳定预览 / 导出”

---

## 当前结论

这套方法到现在，已经被 3 类对象验证过：

- 方法本身：项目门面生成器
- 主题工作流仓库：creator-publish-workflow
- 理论型项目：Ferry

还没完全覆盖的类型包括：

- 纯工程项目
- 强结果导向的小产品
- 需要多平台同步分发的项目

---

## 一句话结论

案例库的作用，不是证明“每次都成功”，而是：

**让这套方法有真实落地样本，而不是只停在规则层。**
