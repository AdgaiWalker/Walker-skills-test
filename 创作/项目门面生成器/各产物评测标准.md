# 各产物评测标准

这份文档不负责生成产物，而是负责给产物过审。

它的目标不是挑刺，而是让这条链上的不同产物有一致的放行标准：

- README
- X 发帖
- 信息卡
- SVG 视觉门面

一句话理解：

`不是做完就算完成，而是通过才继续。`

---

## 最高评测原则

先过一个人，再过规则。

这份文档里的客观检查，只能保证产物没有明显硬伤；  
真正决定它能不能进入下一步的，是一个更高层的问题：

**如果我是人，我会不会看它？**

也就是说：

- 客观层：检查信息、结构、链接、版式有没有明显问题
- 主观层：判断它有没有停留感、理解感和继续看下去的意愿

默认顺序：

1. 先问：我会不会看它？
2. 再问：它有没有明显错误？
3. 最后决定：继续、修正，还是打回

一句话收束：

**客观检查是底线，人眼停留是核心，主观放行是最终标准。**

---

## 通用分级

所有产物统一用三种结果：

- `绿色`：通过，可以进入下一步
- `黄色`：基本可用，但先自动修一轮再复评
- `红色`：方向不对、信息不足或风险过高，先停下来人工确认

默认原则：

- 绿色：继续自动链路
- 黄色：允许自动修正一次
- 红色：停止自动推进

## README 评测卡

### 第一问

- 我愿不愿继续往下读？
- 第一屏会不会把我劝退？
- 我能不能马上知道这是不是给我用的？

### 评什么

- 是否先回答目标读者最关心的问题
- 是否写对仓库层级与项目类型
- 安装、使用、示例是否可执行
- 是否有明显遗漏、冲突或过度承诺
- 若为正式发布，视觉门面需求是否被识别

### 绿色

- 第一屏清楚回答：这是什么、对谁有用、怎么开始
- 结构符合当前仓库层级
- 安装 / 使用 / 示例能直接用
- 没有明显冲突
- 若进入发布档位，视觉门面处理策略清楚

### 黄色

- 主体方向对，但第一屏偏慢或偏乱
- 有一两处关键遗漏，例如安装、示例、截图
- 结构能看，但不够像正式页面
- 视觉门面有需求，但只补了一半

### 红色

- 读者看完还是不知道这是什么
- 层级写错，例如把集合仓库写成单项目
- 关键使用信息缺失
- 内容明显脱离项目实际
- 发布档位却没有明确视觉 / LICENSE / Git 边界

### 常见失败点

- 把 README 写成作者自言自语
- 第一屏塞理念，不先给入口
- 安装和示例只写名字，不可执行
- 过度承诺，仓库里没有的能力也写上

## X 发帖评测卡

### 第一问

- 前两行会不会让我停下来？
- 我会不会想看下一条跟帖？
- 我会不会觉得这像一个人在说话，而不是一段说明书？

### 评什么

- 前两行能不能停住人
- 是不是一眼知道在发什么
- 主帖是不是太满
- 结果 / 动机 / 对比三种重心是否选对
- 链接、配图、跟帖分工是否合理

### 绿色

- 前两行有钩子
- 用户能立刻知道这是什么项目 / skill / 仓库
- 主帖不过满，读起来像人写的
- 结果帖、动机帖或对比帖判断正确
- 链接位置合理，通常在第一条跟帖

### 黄色

- 有信息，但钩子弱
- 主帖能看懂，但略满
- 动机和结果都有，但顺序不够稳
- 跟帖能补清，但主帖本身还不够利落

### 红色

- 看完主帖不知道在发什么
- 一上来全是抽象感受，没有落点
- 主帖像 README 摘抄或官宣文
- 链接过早塞满主帖
- 图文重复，或者图和文案各说各话

### 常见失败点

- 把“为什么做”写成自我感动
- 把“结果展示帖”写成空洞理念帖
- 为了完整，把所有信息都堆进主帖
- 不分主帖和跟帖的职责

## 信息卡评测卡

### 第一问

- 我会不会一眼先看主标题？
- 这张图会不会让我愿意停 2 秒？
- 它看起来是信息卡，还是一堆字被塞进盒子里？

### 评什么

- 一张图里，信息密度和阅读路径是否平衡
- 标题、主判断、支持模块有没有层级
- 比例是否真正影响布局，而不是简单缩放
- 文字在移动端是否可读
- 画面是否有明显留白失衡或空盒子

### 绿色

- 一眼能看出主标题和主判断
- 至少一个重模块、一个中模块、一个轻模块
- 不同比例下阅读路径合理
- 18px 左右正文仍清楚可读
- 画面没有大面积无意义留白

### 黄色

- 信息基本完整，但层级不够强
- 画面可读，但下半区偏空
- 比例适配基本到位，但某些模块过轻
- 标题区和正文区分得不够开

### 红色

- 看起来像把长文硬塞进一张图
- 所有模块等权，读者不知道先看哪
- 标题、正文、边栏互相打架
- 下半部分是大块死白
- 移动端文字已经不可读

### 常见失败点

- 把不同内容塞进同样大小的盒子
- 只改比例，不改结构
- 只追求密度，不追求主次
- 标题被挤成竖条，或标题太长压住正文

## SVG 视觉门面评测卡

### 第一问

- 它是不是一个值得看一眼的视觉锚点？
- 不动的时候，我会不会觉得它成立？
- 我会不会因为它的视觉气质而想继续了解这个项目？

### 评什么

- 标题 / 横幅 / 字形是否有明确主视觉作用
- 是否真的适合做 SVG，而不是应交给截图或信息卡
- 结构是否适合拆件
- 动画前后是否都成立
- 是否和项目门面定位一致

### 绿色

- SVG 本体就有独立视觉价值
- 标题短、可拆、可组装
- 不动画时也成立，动画后更强
- 跟项目的定位和语气一致
- 适合作为发布门面的首屏视觉

### 黄色

- 能用，但还像普通字排版
- 结构可拆，但零件层级一般
- 静态成立，动态表现力一般
- 需要再补副标题或更清楚的视觉锚点

### 红色

- 根本不适合用 SVG 处理
- 标题太长、太散、不可拆
- 只是文字截图，不是视觉门面
- 动画和项目定位冲突
- 只有动效，没有静态门面价值

### 常见失败点

- 什么都想做成 SVG
- 标题过长，做完也不能当门面
- 只考虑动画，不考虑静态首屏
- 动画风格和项目气质不一致

---

## 放行规则

### README

- 绿色：可进入视觉门面 / LICENSE / Git
- 黄色：先修再复评
- 红色：停

### X 发帖

- 绿色：可发布
- 黄色：先压短或改重心
- 红色：重写

### 信息卡

- 绿色：可截图 / 可发
- 黄色：先调层级或补结构
- 红色：重做布局

### SVG 视觉门面

- 绿色：可进入动画或作为正式发布视觉
- 黄色：先收主标题或重做拆件
- 红色：不要强行用 SVG，换截图或信息卡

---

## 一句话结论

评测标准的作用不是让每个产物都变漂亮，而是：

**让这条工作流知道什么时候该继续，什么时候该停下来。**
