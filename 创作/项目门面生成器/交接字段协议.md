# 交接字段协议

这份协议不是单个平台的发帖模板，也不是某一个 skill 的内部备注。

它的作用是：

- 让上游 skill 产出一份稳定可复用的交接底稿
- 让下游 skill 直接消费这份底稿，而不是每次重新猜项目
- 让整个工作流从“靠感觉串联”变成“按字段交接”

一句话理解：

`先交接字段，再交接产物。`

---

## 上层指导理论

这份协议本身不是凭空长出来的。

如果往上追，它受这一层问题意识约束：

- 人的意图怎样稳定地进入现实产出
- 一次对话怎样沉淀成下一次还能复用的方法
- 人的判断、AI 的执行、系统的记忆怎样接成一条链

这套更上层的指导理论，可以继续参考：

- [Ferry](https://github.com/AdgaiWalker/Ferry)

在这条关系里：

- `Ferry` 提供上层的人机协作问题意识
- `项目门面生成器` 把这套问题意识落到“门面整理”这个具体动作上
- `交接字段协议` 再把门面整理后的结果，稳定交给下游 skill

---

## 协议目标

这份协议主要解决 4 个问题：

1. 上游 skill 已经整理过的信息，不能在下游再丢失
2. 下游 skill 不应该每次重新猜“这是什么、为什么做、给谁用”
3. 同一个项目在 README、X、小红书、信息卡、SVG 门面之间，应共享同一份核心底稿
4. 字段缺失时，要知道由谁补、何时补、补到什么程度

---

## 适用范围

当前适用这些链路：

- `项目门面生成器` -> `秋知X推文发帖生成器`
- `项目门面生成器` -> `信息卡截图生成器`
- `项目门面生成器` -> `SVG组装动画器`
- 未来扩展到小红书、抖音、B站、知乎等平台适配器

默认原则：

- `项目门面生成器` 是主要上游产出方
- 发帖类、视觉类、平台适配类 skill 是主要消费方

---

## 最小交接字段

以下字段构成最小可用底稿。

| 字段 | 作用 | 必填 | 优先产出方 | 典型消费方 |
| --- | --- | --- | --- | --- |
| `项目名` | 统一名称 | 是 | 项目门面生成器 | 全部 |
| `一句话定位` | 一句话说清这是什么 | 是 | 项目门面生成器 | README / X / 小红书 / 信息卡 |
| `为什么做` | 创作者动机或项目动机 | 是 | 项目门面生成器 | X / 小红书 / 知乎 |
| `解决什么问题` | 明确价值 | 是 | 项目门面生成器 | README / 发帖 / 信息卡 |
| `对谁有用` | 目标读者 / 用户 | 是 | 项目门面生成器 | README / 发帖 / 信息卡 |
| `核心亮点` | 3 个以内核心卖点 | 是 | 项目门面生成器 | README / 发帖 / 信息卡 / 视觉门面 |
| `使用方式` | 安装、调用、进入方式 | 是 | 项目门面生成器 | README / 发帖 |
| `主链接` | 主入口链接 | 是 | 项目门面生成器 | 发帖 / 信息卡 |
| `推荐截图` | 最适合对外展示的图 | 是 | 项目门面生成器 | X / 小红书 / 信息卡 |
| `当前状态` | 已完成 / 迭代中 / 首次开源 / 新版本 | 否 | 项目门面生成器 | 发帖 |
| `最想强调的一句话` | 这次传播最想让人记住的点 | 是 | 项目门面生成器或人工补充 | X / 小红书 / 信息卡 |
| `当前发布目标` | 传播 / 引流 / 收反馈 / 介绍更新 | 是 | 项目门面生成器或人工补充 | 发帖 / 平台适配 |

## 扩展交接字段

当项目进入正式发布或视觉门面阶段时，再补这些字段：

| 字段 | 作用 | 必填 | 优先产出方 | 典型消费方 |
| --- | --- | --- | --- | --- |
| `发布档位` | 快速 / 标准 / 发布 | 否 | 项目门面生成器 | 全部 |
| `视觉意图` | 是否明确需要视觉门面 | 否 | 人工决策 / 项目门面生成器记录 | 视觉类 skill |
| `可复用视觉素材` | Logo / 标题字形 / 横幅图 / 封面图 | 否 | 项目门面生成器调查 | SVG / 信息卡 |
| `视觉主标题` | 适合做封面或标题动画的短标题 | 否 | 项目门面生成器 / 人工补充 | SVG / 信息卡 |
| `视觉副标题` | 适合做封面补充说明的短句 | 否 | 项目门面生成器 | SVG / 信息卡 |
| `推荐平台顺序` | 先发哪里，后发哪里 | 否 | 平台总控或人工 | 平台适配器 |
| `推荐传播角度` | 动机 / 结果 / 对比 / 观点 | 否 | 发帖 skill | 平台适配器 |

## 产出责任

### 上游产出方

默认由 `项目门面生成器` 负责尽量补齐这些字段：

- 项目名
- 一句话定位
- 为什么做
- 解决什么问题
- 对谁有用
- 核心亮点
- 使用方式
- 主链接
- 推荐截图
- 当前状态

如果无法稳定推断：

- 先从 README、代码、截图、文档中补
- 再最小化向用户确认

### 下游消费方

默认由下游 skill 直接消费这些字段，而不是重新猜。

例如：

- `秋知X推文发帖生成器`
  重点吃：`一句话定位 / 为什么做 / 最想强调的一句话 / 主链接 / 推荐截图`

- `信息卡截图生成器`
  重点吃：`一句话定位 / 解决什么问题 / 对谁有用 / 核心亮点 / 推荐截图`

- `SVG组装动画器`
  重点吃：`视觉主标题 / 视觉副标题 / 可复用视觉素材 / 项目名`

---

## 缺失补齐规则

字段缺失时，不是一律停下问用户。

优先顺序：

1. 先从已有上下文补
2. 再从仓库文件补
3. 再从现有截图、文档、README 补
4. 只有会明显影响产物方向时，才最小化向用户确认

规则：

- 能从项目中稳定推断的，不问用户
- 不能稳定推断但不影响主流程的，允许留空
- 不能稳定推断且会直接影响产物方向的，才追问

---

## 交接时机

推荐交接时机如下：

### 1. README 初稿完成后

先形成最小底稿：

- 项目名
- 一句话定位
- 为什么做
- 解决什么问题
- 对谁有用

### 2. README 评测通过后

再补全可对外分发字段：

- 核心亮点
- 使用方式
- 主链接
- 推荐截图
- 最想强调的一句话
- 当前发布目标

### 3. 正式发布前

若进入发布档位，再补视觉字段：

- 视觉意图
- 可复用视觉素材
- 视觉主标题
- 视觉副标题

## 最低交付标准

只要要进入下游 skill，至少应有：

- 项目名
- 一句话定位
- 为什么做
- 解决什么问题
- 对谁有用
- 主链接

如果进入视觉类 skill，再额外至少有：

- 视觉意图
- 推荐截图，或可复用视觉素材之一

---

## 一句话结论

这个协议的核心不是“多写字段”，而是：

**让每个下游 skill 都直接吃上游已经确认过的信息，而不是重新猜。**
