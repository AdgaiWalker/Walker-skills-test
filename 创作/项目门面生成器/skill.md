# 项目门面生成器

基于充分上下文，生成项目对外展示所需的 README 与收尾材料。

## 执行原则

- 人只负责决策，AI 负责分析、生成、整理与执行
- 不问 AI 自己能判断的问题
- 优先使用当前对话与工作区中的已有上下文
- 若上下文不足，先补齐最小必要上下文，再继续生成
- 仅在关键信息缺失时，用少量选项提问
- README 完成后，不立刻结束，要继续给出后续建议

## 适用前提

启用后，若当前已存在足够上下文，可直接进入生成阶段，例如：

- 已读取项目代码、文档或已有 README
- 已理解项目用途、受众或用户目标

若上下文不足，则先在当前工作区补齐最小必要上下文，再继续生成。

## 执行流程

1. 判断当前上下文是否充分
2. 若不足，先读取工作区关键文件补齐最小必要上下文
3. 基于上下文判断项目类型
4. 生成或优化 README 作为主产物
5. 检查遗漏项
6. README 完成后，给出 LICENSE / Git 后续建议

### 最小必要上下文

当上下文不足时，优先读取能快速建立项目认知的关键文件，例如：

- 顶层 `README.md`
- 依赖或项目定义文件，如 `package.json`、`pyproject.toml`、`requirements.txt`、`Cargo.toml`、`go.mod`
- 主要源码目录
- 文档目录
- 示例目录
- 配置文件

只读取建立最小必要上下文所需的关键文件，避免无差别扫描整个项目。

## README 规则

按项目类型选择最小必要结构，不堆无关章节。

常见类型：

- 库 / 框架
- CLI 工具
- 网站 / 应用
- Skills
- 理论框架
- 文档 / 教程

常见内容：

- 标题 + 一句话介绍
- 安装或使用方法
- 核心亮点或示例
- API / FAQ / 应用场景，仅在需要时加入
- License 说明，仅在已明确或后续需要补充时加入
- 若目标是 Skills，默认补上安装方式与最短调用示例

如项目偏正式发布、品牌展示或社区传播，可优先使用“封面式开头”。

### 封面式开头模板

适用：正式发布、开源项目首页、需要更强展示感的 README。

推荐结构：

    横幅图或品牌图
    H1 标题
    按钮式徽章行
    1 到 2 段高密度介绍

开头规则：

- 优先使用已有横幅图、Logo、截图或品牌素材
- 若项目无现成图片，不虚构外链图片，可退化为标题 + 徽章 + 简介
- 徽章优先放核心跳转，如 Docs、官网、License、社区、发布方
- 开头文案要短，先讲项目是什么，再讲它为什么值得用
- 若采用封面式开头，正文第一屏不要再堆大段背景介绍

## 提问规则

仅在缺少关键决策时，用少量选项提问。

可补问：

- 目标用户：开发者 / 决策者 / 普通用户 / AI 用户 / 思考者
- 风格：极简 / 标准

如上下文已足够，直接生成，不再追问。

## 填充原则

- 如无必要勿增实体
- 保留最小可行示例
- 语言简洁，直击痛点
- 优先保留已有信息，再做重构与美化
- 若许可证尚未明确，不擅自写死具体许可证类型
- 若目标是 Skills，安装方式优先从上下文直接推断，不把本可推断的信息继续丢给用户决策

## Skills 安装规则

当目标是 Skills 类型时，README 默认包含 `安装` 部分，不要省略。

优先输出两种安装方式：

1. 命令安装
2. 让 AI 帮你安装

### 方式 1：命令安装

若上下文中已能判断仓库地址与 skill 名称，直接生成可复制命令：

```bash
npx skills add <owner/repo> --skill <skill-name>
```

### 方式 2：让 AI 帮你安装

提供一段可以直接贴给 AI 的提示词，默认包含：

- 仓库链接
- skill 名称
- 优先使用命令安装
- 安装完成后告知最短调用方式

参考结构：

```text
帮我安装这个 skill：
<repo-url>

skill 名称：<skill-name>

请帮我用合适的方式完成安装。
如果支持命令安装，优先使用：
npx skills add <owner/repo> --skill <skill-name>

安装完成后，告诉我最短怎么调用它。
```

规则：

- 仓库地址、skill 名称、安装命令能从上下文推断时，直接补全，不再追问
- 只有仓库地址或 skill 名称无法判断时，才最小化补问
- 给人看的 README 优先保证“能复制、能看懂、能直接交给 AI”

## 美化规则

已有 README 时：

1. 保留核心信息
2. 调整结构和层级
3. 按所选风格统一视觉层次
4. 补齐必要但缺失的内容

风格：

- 极简：少徽章、少装饰、正文优先
- 标准：适量徽章、清晰分节、表格与引用适度使用
- 标准风格在合适时可使用“封面式开头”

## README 完成后的后续建议

README 完成后，不要直接结束。

AI 先自行检查：

- 项目名称与一句话介绍是否完整
- 安装方式是否缺失
- 使用方法是否缺失
- 示例、截图、演示入口是否缺失
- 是否已有 LICENSE
- README 中是否已有 License 说明
- 当前目录是否为 Git 仓库
- 是否已配置远程仓库
- 是否存在可提交改动

然后一次性给出后续建议，让用户只做决策。

示例表达：

README 已完成。建议继续处理：

- LICENSE：补充或更新许可证
- Git：提交并按需推送本次改动

用户只需决定：

- 两者都处理
- 只处理 LICENSE
- 只处理 Git
- 都不处理

## LICENSE 规则

LICENSE 与 Git 是 README 完成后的两个独立动作，可单独执行，也可同时执行。

若需处理 LICENSE：

- 优先基于上下文推荐，而不是要求用户自行判断
- 常见选项仅保留：
- MIT：宽松，适合大多数项目
- Apache-2.0：宽松，含专利授权
- GPL-3.0：衍生发布需继续开源
- 暂不添加 LICENSE：保留版权，不主动授权

用户确认后：

- 生成或更新 LICENSE 文件
- 同步更新 README 中的 License 说明

## Git 规则

若需处理 Git：

- 检查当前目录是否为 Git 仓库
- 检查是否存在远程仓库
- 检查是否存在可提交改动
- 在用户确认后，再执行提交或推送

## 输出

- 完整流程：README + 后续建议
- 美化 README：只优化 README
- 优化 README：重构内容并美化
- 格式：Markdown，可直接使用
